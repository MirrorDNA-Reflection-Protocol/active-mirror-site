# Active Mirror v0.1 Lock

Date: 2026-06-22
Status: Canonical product lock for the public site and browser workspace

## Product Category

Public category:

```text
Private AI workspace for important decisions.
```

Technical category:

```text
Private decision workspace with receipts, approved memory, and decision history.
```

Do not lead public copy with MirrorDNA, MirrorGraph, abstract decision-memory language, symbolic continuity, cognitive timeline, full OS language, or other architecture-first terms. Those belong only in technical, protocol, investor, or architecture contexts after the basic product is clear.

## Public Promise

```text
AI that helps you decide, shows its work, and remembers only what you approve.
```

The sharper proof-led homepage line can remain:

```text
The AI that proves it.
```

## Core Loop

```text
Important decision or messy workflow
-> Thinking Session
-> Structured Reflection
-> Generated Artifact
-> Receipt
-> Memory Candidate Review
-> Approved Private Context
-> Later Outcome Review
```

Every product surface should serve this loop.

## v0.1 User Job

A user should be able to bring one real decision, problem, transcript, meeting note, research thread, launch plan, or messy thought and leave with:

- a clearer framing of the issue;
- visible uncertainties, risks, assumptions, and blind spots;
- one useful artifact, such as a decision summary, strategy note, action plan, meeting prep, research brief, or reflection note;
- a receipt showing context used, context excluded, route taken, and memory decision;
- a choice to approve, edit, save temporarily, export, forget, or block memory.

## What Makes Active Mirror Different

Reflection alone is not the moat.

The moat is:

```text
Reviewable reflection with approved memory.
```

Competitors can remember facts, preferences, documents, repos, and conversation history. Active Mirror must make the reasoning and memory decision inspectable before it becomes continuity.

## v0.1 Must-Haves

- Browser-first workspace.
- Context packet preview before sensitive routes.
- Boundary or trust mode per session.
- Generated summary, plan, board, brief, or other useful work output, not only chat text.
- Receipt visible for serious output.
- Follow-up questions capped at three, asked only when they change the decision, risk, or memory outcome.
- Memory candidate review with no preselected approval.
- Browser vault or local persistence where available.
- Exportable artifact or receipt.
- Deterministic local fallback when model routes are unavailable.
- Clear public language that avoids overclaiming live model or infrastructure status.

## v0.1 Must-Not-Haves

- No silent memory writes.
- No frontend provider secrets.
- No autonomous external actions.
- No full OS replacement claim.
- No per-user model training promise.
- No public client/institutional claims without explicit sign-off.
- No Gemini as the core reflection default.
- No vague "AI companion", "AI twin", "second brain", or generic chatbot positioning.
- No provider route claim unless the gateway proves it live.

## Model Routing Policy

Default product posture:

- GPT-first for reasoning through decisions.
- Claude for critique, chat polish, and receipt review.
- Gemini for media, images, video, and multimodal work.
- Local deterministic fallback when the gateway or provider is unavailable.

Providers are helpers behind gates. They are not the product.

## Trust Contract

For serious outputs, the receipt must answer:

1. Why this was suggested.
2. What context was used.
3. What context was excluded.
4. Which route or model path was taken.
5. What memory decision was made or proposed.

If a claim is unsupported, stale, inferred, or uncertain, it must be marked as such rather than promoted as fact.

## Evaluation Gates

Before calling v0.1 production-ready, test:

- unsupported claims;
- sycophancy and over-agreement;
- bad memory proposals;
- stale memory;
- sensitive inference;
- contradiction handling;
- receipt completeness;
- mobile usability;
- provider fallback behavior;
- export integrity.

Alpha targets:

- Important answer sentences should have receipt support where possible.
- Sensitive or interpretive memory should require explicit review.
- Durable memory false positives should be rare enough that users trust the review step.

## Drift Stop

If the product discussion drifts toward browser OS, Siri replacement, full agent swarm, local daemon, phone mesh, autonomous workflows, or abstract theory language before the workspace loop is reliable, return to:

```text
Important decision -> artifact -> receipt -> approved memory.
```
