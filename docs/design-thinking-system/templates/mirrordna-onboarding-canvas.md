# MirrorDNA Onboarding Canvas

Use this to create a user's first mirror without over-collecting personal context.

## Purpose

Clarify the user's working identity, current goal, preferred style, boundaries, and memory defaults.

## Inputs

- One thing the user wants to move forward.
- Current working context.
- Boundaries: what not to store, share, infer, or act on.
- Preferred output shape.
- Allowed tools.
- Memory preference.

## Canvas

| Field | Prompt | Answer |
| --- | --- | --- |
| Current goal | What are you trying to move forward? |  |
| Success signal | What would count as useful progress today? |  |
| Current friction | What is making this harder than it should be? |  |
| Preferred help | Decide, research, draft, organize, review, reset, or build? |  |
| Output shape | Plan, document, table, visual, code, checklist, or artifact? |  |
| Boundary | What should Active Mirror never store, share, or assume? |  |
| Tool permission | What tools may be used without asking again? |  |
| Memory default | What may be remembered as a reusable default? |  |
| Reset phrase | What should stop drift and return to the grounded goal? |  |

## Role Critique

| Role | Check |
| --- | --- |
| UX | Can the user understand and change the mirror later? |
| PM | Does this map to a real job, not a vague personality profile? |
| Engineer | Can these preferences become enforceable defaults? |
| Privacy | Are sensitive fields minimized and revocable? |

## Receipt Fields

- Goal captured.
- Boundary captured.
- Memory decision.
- Tools allowed.
- Defaults created.
- Defaults rejected.
- Next step.

## Exit Criteria

- The mirror can reflect one live task.
- The user can see what is saved and what is not.
- The first receipt is generated.
